\section{Introduction}
% ============================================================

Ce projet vise l’implémentation d’un processus métier complet d’évaluation de demandes de prêt immobilier en architecture microservices.

L’objectif est de concevoir un système distribué respectant :

\begin{itemize}
\item La séparation des responsabilités (Single Responsibility Principle)
\item L’asynchronisme pour les traitements longs
\item La communication événementielle fiable
\item La cohérence transactionnelle
\item La tolérance aux pannes
\end{itemize}

Le système repose sur FastAPI, RabbitMQ, Celery et PostgreSQL.

\section{Architecture Détaillée du Système}

\subsection{Frontières et Responsabilités des Microservices}

Chaque microservice du système possède une responsabilité unique et clairement définie selon le principe de séparation des préoccupations.

\subsubsection{Service Demandes}

\textbf{Responsabilité :} Gestion du cycle de vie des demandes de prêt immobilier.

\textbf{Fonctionnalités :}
\begin{itemize}
    \item Création et validation des demandes
    \item Stockage des informations du client
    \item Agrégation des résultats des évaluations
    \item Mise à jour du statut global
    \item Exposition des endpoints REST pour consultation
\end{itemize}

\textbf{Base de données :} Table \texttt{demandes} avec historisation complète.

\subsubsection{Service Vérification Crédit}

\textbf{Responsabilité :} Évaluation de la solvabilité du demandeur.

\textbf{Fonctionnalités :}
\begin{itemize}
    \item Calcul du score de crédit
    \item Vérification des antécédents financiers
    \item Calcul du ratio dette/revenu
    \item Détection des dettes existantes
    \item Publication de l'événement \texttt{credit.evaluated}
\end{itemize}

\textbf{Traitement :} Asynchrone via Celery avec durée typique de 3 à 8 secondes.

\subsubsection{Service Évaluation Bien}

\textbf{Responsabilité :} Estimation de la valeur du bien immobilier.

\textbf{Fonctionnalités :}
\begin{itemize}
    \item Estimation de la valeur marchande
    \item Vérification de la valeur cadastrale
    \item Évaluation de l'état du bien
    \item Analyse de la zone géographique
    \item Calcul du ratio prêt/valeur (LTV)
    \item Publication de l'événement \texttt{bien.evaluated}
\end{itemize}

\textbf{Traitement :} Asynchrone via Celery avec durée typique de 5 à 10 secondes.

\subsubsection{Service Décision}

\textbf{Responsabilité :} Prise de décision finale basée sur les règles métier.

\textbf{Fonctionnalités :}
\begin{itemize}
    \item Agrégation des évaluations crédit et bien
    \item Application des règles DMN (Decision Model Notation)
    \item Calcul du score global
    \item Détermination du taux d'intérêt
    \item Génération des conditions supplémentaires
    \item Publication de l'événement \texttt{decision.made}
\end{itemize}

\textbf{Règles métier :} Implémentation des seuils de décision automatisée.

\subsubsection{Service Notification}

\textbf{Responsabilité :} Communication des résultats au client.

\textbf{Fonctionnalités :}
\begin{itemize}
    \item Envoi d'emails de notification
    \item Mise à jour des interfaces temps réel (WebSocket/SSE)
    \item Archivage des notifications envoyées
    \item Gestion des préférences de communication
\end{itemize}

\subsection{Principe d'Isolation des Bases de Données}

Chaque microservice maintient sa propre base de données logiquement isolée, conformément au pattern \textit{Database per Service}.

\textbf{Avantages de cette approche :}

\begin{enumerate}
    \item \textbf{Couplage faible :} Aucun service n'accède directement aux données d'un autre service
    \item \textbf{Évolution indépendante :} Les schémas peuvent évoluer sans coordination
    \item \textbf{Choix technologique libre :} Possibilité d'utiliser des SGBD différents par service
    \item \textbf{Scalabilité ciblée :} Chaque base peut être dimensionnée indépendamment
\end{enumerate}

\textbf{Implémentation actuelle :}

\begin{itemize}
    \item PostgreSQL 15 comme SGBD unique
    \item Bases logiques séparées par service
    \item Schémas isolés (pas de jointures inter-services)
    \item Communication via événements uniquement
\end{itemize}

\subsection{Architecture Événementielle Asynchrone}

Le système adopte le pattern \textit{Event-Driven Architecture} pour la communication inter-services.

\subsubsection{Principes Architecturaux}

\begin{enumerate}
    \item \textbf{Publication d'événements :} Chaque service publie des événements métier significatifs
    \item \textbf{Abonnement sélectif :} Les services s'abonnent uniquement aux événements pertinents
    \item \textbf{Couplage temporel faible :} Émetteur et consommateur sont découplés dans le temps
    \item \textbf{Traitement asynchrone :} Pas d'attente synchrone de réponse
\end{enumerate}

\subsubsection{Flux de Communication}

Le flux complet d'une demande de prêt suit le schéma suivant :

\begin{enumerate}
    \item \textbf{Initialisation :} Création de la demande via REST API
    \item \textbf{Évaluation crédit :} Tâche Celery + publication \texttt{credit.evaluated}
    \item \textbf{Évaluation bien :} Tâche Celery + publication \texttt{bien.evaluated}
    \item \textbf{Décision :} Règles métier + publication \texttt{decision.made}
    \item \textbf{Notification :} Envoi au client + publication \texttt{notification.sent}
\end{enumerate}

Chaque étape est autonome et peut être rejouée en cas d'échec.

\section{Configuration Détaillée de Celery}
% ============================================================

\subsection{Architecture Broker et Backend}

Celery est configuré avec une architecture standard composée de :

\begin{itemize}
    \item \textbf{Broker :} RabbitMQ (gestion des files de tâches)
    \item \textbf{Backend :} Redis (stockage des résultats)
    \item \textbf{Workers :} Processus Python exécutant les tâches
\end{itemize}

\subsection{Configuration du Broker}

\begin{lstlisting}[language=Python]
broker_url = 'amqp://guest:guest@rabbitmq:5672//'
broker_transport_options = {
    'visibility_timeout': 3600,  # 1 heure
    'max_retries': 5,
    'interval_start': 0,
    'interval_step': 0.2,
    'interval_max': 1,
}
\end{lstlisting}

\textbf{Paramètres clés :}

\begin{itemize}
    \item \texttt{visibility\_timeout} : Durée maximale d'exécution d'une tâche avant redelivery
    \item \texttt{max\_retries} : Nombre de tentatives de connexion au broker
    \item Intervalle exponentiel pour les reconnexions
\end{itemize}

\subsection{Configuration du Backend de Résultats}

\begin{lstlisting}[language=Python]
result_backend = 'redis://redis:6379/0'
result_expires = 3600  # 1 heure
result_persistent = True
\end{lstlisting}

Les résultats sont stockés dans Redis pour permettre :
\begin{itemize}
    \item La récupération asynchrone des résultats
    \item Le suivi de l'état des tâches
    \item La persistance temporaire (1 heure)
\end{itemize}

\subsection{Stratégie de Retry et Gestion d'Erreurs}

Chaque tâche Celery est configurée avec une stratégie de retry exponentielle.

\begin{lstlisting}[language=Python]
@celery_app.task(
    bind=True,
    max_retries=5,
    default_retry_delay=60,
    autoretry_for=(Exception,),
    retry_backoff=True,
    retry_backoff_max=600,
    retry_jitter=True,
    acks_late=True,
    reject_on_worker_lost=True
)
def evaluate_credit(self, demande_id: str):
    try:
        # Traitement de la tâche
        pass
    except Exception as exc:
        raise self.retry(exc=exc, countdown=60)
\end{lstlisting}

\textbf{Paramètres de fiabilité :}

\begin{table}[H]
\centering
\begin{tabular}{lp{8cm}}
\toprule
\textbf{Paramètre} & \textbf{Description} \\
\midrule
\texttt{max\_retries} & Nombre maximum de tentatives (5) \\
\texttt{retry\_backoff} & Délai exponentiel entre retries \\
\texttt{retry\_backoff\_max} & Délai maximum de 600 secondes \\
\texttt{retry\_jitter} & Ajout d'aléa pour éviter la synchronisation \\
\texttt{acks\_late} & ACK après exécution (pas avant) \\
\texttt{reject\_on\_worker\_lost} & Redelivery si worker crash \\
\bottomrule
\end{tabular}
\caption{Paramètres de fiabilité des tâches Celery}
\end{table}

\textbf{Séquence de retry :}

\begin{verbatim}
Tentative 1: Immédiate
Tentative 2: +60s
Tentative 3: +120s
Tentative 4: +240s
Tentative 5: +480s
Tentative 6: +600s (max)
→ Dead Letter Queue si échec final
\end{verbatim}

\subsection{Scalabilité des Workers}

Les workers Celery peuvent être scalés horizontalement sans modification du code.

\textbf{Mécanisme de distribution :}

\begin{enumerate}

\end{enumerate}

\textbf{Dimensionnement typique :}

\begin{itemize}
    \item 1 worker pour 100 demandes/jour
    \item 4 workers pour 1000 demandes/jour
    \item 10+ workers pour charge élevée
\end{itemize}

\section{Configuration Avancée de RabbitMQ}
% ============================================================

\subsection{Architecture Exchange et Queues}

RabbitMQ utilise un exchange de type \texttt{TOPIC} pour le routage flexible des messages.

\begin{lstlisting}[language=Python]
exchange = await channel.declare_exchange(
    'pret_immobilier_events',
    ExchangeType.TOPIC,
    durable=True
)
\end{lstlisting}

\textbf{Avantages du type TOPIC :}

\begin{itemize}
    \item Routage basé sur des patterns (ex: \texttt{credit.*}, \texttt{*.evaluated})
    \item Flexibilité pour ajouter de nouveaux consommateurs
    \item Support des wildcards (\texttt{*} et \texttt{\#})
\end{itemize}

\subsection{Durabilité et Persistance}

Tous les éléments critiques sont configurés comme durables.

\begin{lstlisting}[language=Python]
# Déclaration de queue durable
queue = await channel.declare_queue(
    'demande_events_queue',
    durable=True,
    arguments={
        'x-message-ttl': 86400000,  # 24 heures
        'x-max-length': 10000,
        'x-overflow': 'reject-publish'
    }
)
\end{lstlisting}

\textbf{Garanties de persistance :}

\begin{enumerate}
    \item \textbf{Exchange durable :} Survit au redémarrage de RabbitMQ
    \item \textbf{Queue durable :} Conservée après redémarrage
    \item \textbf{Messages persistants :} Écrits sur disque
    \item \textbf{Acknowledgments :} Pas de perte en cas de crash worker
\end{enumerate}

\subsection{Dead Letter Queue (DLQ)}

Une Dead Letter Queue est configurée pour chaque queue principale.

\begin{lstlisting}[language=Python]
# Configuration de la DLQ
dlx_exchange = await channel.declare_exchange(
    'pret_immobilier_events.dlx',
    ExchangeType.TOPIC,
    durable=True
)

dlq = await channel.declare_queue(
    'demande_events_queue.dlq',
    durable=True
)

# Liaison queue principale -> DLQ
main_queue = await channel.declare_queue(
    'demande_events_queue',
    durable=True,
    arguments={
        'x-dead-letter-exchange': 'pret_immobilier_events.dlx',
        'x-message-ttl': 60000,  # Retry après 60s
    }
)
\end{lstlisting}

\textbf{Stratégie de retry avec DLQ :}

\begin{verbatim}
Message initial
    ↓
Traitement échoue
    ↓
Reject + requeue=False
    ↓
Envoi vers retry queue (TTL 60s)

Expiration du TTL
    ↓
Retour vers queue principale
    ↓
Nouvelle tentative
    ↓
Si échec après 5 retries → DLQ finale
\end{verbatim}

\subsection{Acknowledgment et Garantie de Livraison}

Le système utilise le mode \texttt{acks\_late} pour garantir la fiabilité.

\begin{lstlisting}[language=Python]
async with message.process(requeue=False):
    try:
        event = parse_event(message.body)
        await handle_event(event)
        # ACK automatique si succès
    except Exception as e:
        logger.error(f"Erreur: {e}")
        raise  # REJECT automatique si exception
\end{lstlisting}

\textbf{Comportement du context manager \texttt{message.process()} :}

\begin{table}[H]
\centering
\begin{tabular}{lll}
\toprule
\textbf{Scénario} & \textbf{Action} & \textbf{Résultat} \\
\midrule
Traitement réussi & ACK & Message supprimé \\
Exception levée & REJECT & Message vers DLX \\
Worker crash & Timeout & Redelivery automatique \\
Network failure & Timeout & Redelivery automatique \\
\bottomrule
\end{tabular}
\caption{Comportement de l'acknowledgment selon le scénario}
\end{table}

\textbf{Paramètre \texttt{requeue=False} :}

Ce paramètre est critique pour éviter les boucles infinies :
\begin{itemize}
    \item \texttt{requeue=True} : Remet le message en tête de queue (risque de boucle)
    \item \texttt{requeue=False} : Envoie vers Dead Letter Exchange (comportement souhaité)
\end{itemize}

\section{Cohérence Transactionnelle et Idempotence}
% ============================================================

Un point critique du projet concerne la garantie d’un traitement exactly-once.

\subsection{Idempotence atomique}

L’idempotence est assurée via PostgreSQL :

\begin{lstlisting}[language=SQL]
INSERT INTO processed_messages (correlation_id, ...)
VALUES (...)
ON CONFLICT (correlation_id) DO NOTHING
RETURNING correlation_id;
\end{lstlisting}

Si l’insertion retourne NULL, le message a déjà été traité.

\subsection{Transaction unique}

Toutes les écritures sont encapsulées dans :

\begin{lstlisting}
async with session.begin():
    ...
\end{lstlisting}

Cela garantit :

\begin{itemize}
\item Atomicité
\item Rollback automatique en cas d'erreur
\item Absence de fenêtre de crash
\end{itemize}

% ============================================================
\section{Sécurité Transactionnelle et Idempotence Avancée}
% ============================================================

\subsection{Problématique du Traitement Exactly-Once}

Dans un système distribué, garantir qu'un message est traité exactement une fois est un défi majeur.

\textbf{Scénarios problématiques :}

\begin{enumerate}
    \item \textbf{Double delivery :} RabbitMQ livre le message deux fois (redelivery)
    \item \textbf{Workers concurrents :} Deux workers consomment le même message
    \item \textbf{Crash avant ACK :} Worker crash après traitement mais avant ACK
    \item \textbf{Network partition :} Connexion DB perdue entre update et ACK
\end{enumerate}

\subsection{Stratégie INSERT ON CONFLICT}

La solution adoptée utilise les capacités atomiques de PostgreSQL.

\subsubsection{Schéma de la Table d'Idempotence}

\begin{lstlisting}[language=SQL]
CREATE TABLE processed_messages (
    correlation_id VARCHAR(255) PRIMARY KEY,
    event_type VARCHAR(100) NOT NULL,
    demande_id VARCHAR(50) NOT NULL,
    processor_service VARCHAR(100) NOT NULL,
    processed_at TIMESTAMP NOT NULL DEFAULT NOW()
);
\end{lstlisting}

La contrainte \texttt{PRIMARY KEY} sur \texttt{correlation\_id} garantit l'unicité.

\subsubsection{Requête Atomique d'Idempotence}

\begin{lstlisting}[language=SQL]
INSERT INTO processed_messages 
    (correlation_id, event_type, demande_id, processor_service)
VALUES 
    (:correlation_id, :event_type, :demande_id, :service)
ON CONFLICT (correlation_id) DO NOTHING
RETURNING correlation_id;
\end{lstlisting}

\textbf{Comportement :}

\begin{itemize}
    \item Si \texttt{correlation\_id} n'existe pas : INSERT réussit, retourne la valeur
    \item Si \texttt{correlation\_id} existe : INSERT échoue silencieusement, retourne NULL
\end{itemize}

Cette opération est \textbf{atomique au niveau de la base de données}, éliminant toute race condition.

\subsubsection{Implémentation dans le Code}

\begin{lstlisting}[language=Python]
async def try_mark_as_processed(
    session, correlation_id, event_type, demande_id, service
) -> bool:
    query = text("""
        INSERT INTO processed_messages 
            (correlation_id, event_type, demande_id, 
             processor_service, processed_at)
        VALUES 
            (:correlation_id, :event_type, :demande_id, 
             :service, NOW())
        ON CONFLICT (correlation_id) DO NOTHING
        RETURNING correlation_id
    """)
    
    result = await session.execute(query, {
        "correlation_id": correlation_id,
        "event_type": event_type,
        "demande_id": demande_id,
        "service": service
    })
    
    inserted = result.scalar_one_or_none()
    return inserted is not None
\end{lstlisting}

\textbf{Retour de la fonction :}
\begin{itemize}
    \item \texttt{True} : Ce worker a gagné la "course", doit traiter le message
    \item \texttt{False} : Un autre worker a déjà traité le message, ignorer
\end{itemize}

\subsection{Importance de la Transaction Unique}

Toutes les opérations d'un event handler sont encapsulées dans une transaction unique.

\begin{lstlisting}[language=Python]
async def handle_credit_evaluated(event: Event):
    async with AsyncSessionLocal() as session:
        async with session.begin():  # BEGIN transaction
            try:
                # Étape 1: Marquer comme traité (atomique)
                should_process = await try_mark_as_processed(
                    session, event.correlation_id, ...
                )
                
                if not should_process:
                    return  # Déjà traité, rollback automatique
                
                # Étape 2: Mettre à jour les données métier
                await update_credit_evaluation(
                    session, event.demande_id, ...
                )
                
                # COMMIT automatique à la sortie du contexte
                
            except Exception as e:
                # ROLLBACK automatique en cas d'exception
                raise
\end{lstlisting}

\textbf{Garanties fournies par \texttt{async with session.begin()} :}

\begin{enumerate}
    \item \textbf{Atomicité :} Toutes les opérations réussissent ou toutes échouent
    \item \textbf{Commit unique :} Un seul COMMIT à la fin du contexte
    \item \textbf{Rollback automatique :} Toute exception déclenche un ROLLBACK
    \item \textbf{Isolation :} Les autres transactions ne voient pas les changements non committés
\end{enumerate}

\subsection{Élimination de la Fenêtre de Crash}

\textbf{Problème avec commits multiples (AVANT refactoring) :}

\begin{verbatim}
T=0.00s: BEGIN transaction
T=0.01s: UPDATE demandes SET score_credit = 720
T=0.02s: COMMIT  ← Données persistées
T=0.03s: ⚠️ CRASH ICI ⚠️
         (mark_as_processed() non exécuté)
T=1.00s: Message redelivered par RabbitMQ
T=1.01s: Traitement en double (score écrasé)
\end{verbatim}

\textbf{Solution avec transaction unique (APRÈS refactoring) :}

\begin{verbatim}
T=0.00s: BEGIN transaction
T=0.01s: INSERT processed_messages (idempotence)
T=0.02s: UPDATE demandes SET score_credit = 720
T=0.03s: ⚠️ CRASH ICI ⚠️
         (COMMIT non exécuté)
         
PostgreSQL: ROLLBACK automatique

T=1.00s: Message redelivered
T=1.01s: INSERT processed_messages → succès
T=1.02s: UPDATE demandes → exécuté
T=1.03s: COMMIT réussi
\end{verbatim}

La fenêtre de crash est éliminée : soit les deux opérations réussissent, soit aucune ne persiste.

\section{Défis Techniques et Solutions Apportées}
% ============================================================

\subsection{Défi 1 : Concurrence et Race Conditions}

\textbf{Problématique :} Avec plusieurs workers consommant les mêmes queues, des race conditions peuvent survenir lors de la vérification d'idempotence.

\textbf{Solution :} Utilisation de \texttt{INSERT ON CONFLICT} pour une vérification atomique garantie par PostgreSQL.

\begin{lstlisting}[language=SQL]
INSERT INTO processed_messages (correlation_id, ...)
VALUES (...)
ON CONFLICT (correlation_id) DO NOTHING
RETURNING correlation_id;
\end{lstlisting}

\subsection{Défi 2 : Frontière de Transaction et Fenêtre de Crash}

\textbf{Problématique :} Avec des commits multiples, une fenêtre de crash existe entre la mise à jour métier et le marquage d'idempotence.

\textbf{Solution :} Encapsulation de toutes les opérations dans une transaction unique.

\begin{lstlisting}[language=Python]
async def handle_event(event):
    async with AsyncSessionLocal() as session:
        async with session.begin():  # Transaction unique
            should_process = await try_mark_as_processed(...)
            if not should_process:
                return
            await update_demande(...)
            # COMMIT automatique à la sortie
\end{lstlisting}

\subsection{Défi 3 : Sécurité de l'Acknowledgment RabbitMQ}

\textbf{Problématique :} L'ordre des opérations entre COMMIT DB et ACK message est critique.

\textbf{Solution :} Utilisation du context manager \texttt{message.process()} qui garantit que le COMMIT s'exécute avant l'ACK.

\begin{lstlisting}[language=Python]
async with message.process(requeue=False):
    async with session.begin():
        # Traitement
        pass
    # COMMIT ici
# ACK automatique après sortie complète du contexte
\end{lstlisting}

\subsection{Défi 4 : Gestion des Dépendances entre Services}

\textbf{Problématique :} Le service Décision nécessite les résultats des services Crédit et Bien.

\textbf{Solution :} Chorégraphie événementielle avec agrégation dans le service Demandes. Cette approche élimine les dépendances directes entre services.

\section{Saga Pattern : Compensation Métier Distribuée}
% ============================================================

\subsection{Problématique de la Cohérence Distribuée}

Dans une architecture microservices, garantir la cohérence transactionnelle entre services indépendants constitue un défi technique majeur.

\textbf{Problème :} Une transaction métier (demande de prêt) implique plusieurs services :

\begin{enumerate}
    \item Service Crédit évalue la solvabilité → \texttt{credit.evaluated}
    \item Service Bien évalue le bien immobilier → \texttt{bien.evaluated}
    \item Service Décision applique les règles métier → \texttt{decision.made}
\end{enumerate}

\textbf{Scénario d'échec partiel :}

Si le Service Décision échoue après que Crédit et Bien ont réussi, le système se retrouve dans un état incohérent : les ressources (score crédit, évaluation bien) sont allouées mais la demande ne peut aboutir.

\textbf{Solution implémentée :} \textit{Saga Pattern avec compensation chorégraphiée}.

\subsection{Architecture Saga Chorégraphiée}

Le système implémente une Saga basée sur la chorégraphie événementielle, sans orchestrateur central.

\subsubsection{Événements de Compensation}

\begin{table}[H]
\centering
\begin{tabular}{lll}
\toprule
\textbf{Événement} & \textbf{Rôle} & \textbf{Émetteur} \\
\midrule
\texttt{compensation.triggered} & Déclencheur Saga & Bien, Décision \\
\texttt{credit.cancelled} & Annulation crédit & Service Crédit \\
\texttt{bien.cancelled} & Annulation bien & Service Bien \\
\bottomrule
\end{tabular}
\caption{Événements de compensation du Saga Pattern}
\end{table}

\textbf{Structure de l'événement \texttt{compensation.triggered} :}

\begin{lstlisting}[language=json]
{
  "event_type": "compensation.triggered",
  "demande_id": "DEM-ABC123",
  "data": {
    "etape_echec": "prise_decision",
    "raison": "Erreur système lors de la décision"
  }
}
\end{lstlisting}

Le champ \texttt{etape\_echec} indique quelle étape a échoué, permettant aux services précédents de déterminer s'ils doivent compenser.

\subsection{Flux de Compensation : Scénario Échec Décision}

\textbf{Séquence d'événements :}

\begin{verbatim}
1. POST /demandes → Demande créée ✓
2. Service Crédit → credit.evaluated ✓
3. Service Bien → bien.evaluated ✓
4. Service Décision → Exception système ✗
   → Publication compensation.triggered
     data: { "etape_echec": "prise_decision" }

5. Service Bien reçoit compensation.triggered
   → Vérifie : etape_echec == "prise_decision" ✓
   → Publie : bien.cancelled

6. Service Crédit reçoit compensation.triggered
   → Vérifie : etape_echec in ["evaluation_bien", 
               "prise_decision"] ✓
   → Publie : credit.cancelled

7. Service Demandes reçoit compensation.triggered
   → UPDATE demandes SET statut = 'REJECTED'
   → Raison : "Compensation - prise_decision échouée"

RÉSULTAT: Rollback distribué complet ✓
\end{verbatim}

\subsection{Implémentation des Handlers}

\subsubsection{Service Crédit}

\begin{lstlisting}[language=Python]
async def handle_compensation(event: Event):
    """Annule l'évaluation crédit si étape ultérieure échoue"""
    etape_echec = event.data.get("etape_echec")
    
    # Compenser si bien ou décision a échoué
    if etape_echec not in ["evaluation_bien", "prise_decision"]:
        return  # Pas de compensation nécessaire
    
    # Compensation métier réelle :
    # - Annuler réservation crédit auprès de l'organisme
    # - Libérer le score calculé
    # - Annuler allocations de ressources
    
    await event_publisher.publish(
        event_type="credit.cancelled",
        demande_id=event.demande_id,
        data={
            "raison": f"Compensation - {etape_echec} échouée"
        }
    )
\end{lstlisting}

\subsubsection{Service Évaluation Bien}

\begin{lstlisting}[language=Python]

    """Annule l'évaluation bien si décision échoue"""
    etape_echec = event.data.get("etape_echec")

        event_type="bien.cancelled",

\subsection{Différence : Retry Technique vs Compensation Métier}

\textbf{Retry Technique (Celery)} :

\begin{itemize}
    \item \textbf{Objectif :} Gérer les erreurs transitoires
    \item \textbf{Scope :} Une seule tâche isolée
    \item \textbf{Action :} Réexécuter la même tâche
    \item \textbf{Exemple :} Timeout connexion DB → retry après 60s
\end{itemize}

\textbf{Compensation Saga (Business)} :

\begin{itemize}
    \item \textbf{Objectif :} Cohérence transactionnelle distribuée
    \item \textbf{Scope :} Toute la chaîne de transaction métier
    \item \textbf{Action :} Rollback des étapes réussies précédentes
    \item \textbf{Exemple :} Décision échoue → annuler crédit ET bien
\end{itemize}

\textbf{Le système combine les deux :}

\begin{enumerate}
    \item Retry technique : 5 tentatives avec backoff exponentiel
    \item Si échec final : Compensation Saga distribuée
\end{enumerate}

\subsection{Garanties Transactionnelles}

\subsubsection{Idempotence Maintenue}

Chaque handler de compensation utilise la même stratégie \texttt{INSERT ON CONFLICT} :

\begin{lstlisting}[language=SQL]
INSERT INTO processed_messages (correlation_id, ...)
VALUES (...)
ON CONFLICT (correlation_id) DO NOTHING
RETURNING correlation_id;
\end{lstlisting}

\textbf{Garantie :} Un événement \texttt{compensation.triggered} avec le même \texttt{correlation\_id} ne sera traité qu'une seule fois par chaque service.

\subsubsection{Transaction Unique}

Toutes les opérations de compensation sont encapsulées dans une transaction unique :

\begin{lstlisting}[language=Python]
async with session.begin():  # Transaction atomique
    should_process = await try_mark_as_processed(...)
    if not should_process:
        return
    await update_demande(...)  # Mise à jour métier
    # COMMIT automatique
\end{lstlisting}

\textbf{Fenêtre de crash éliminée :} Si le worker crash avant COMMIT, toutes les opérations sont rollback.
